## New Action Details

### `metadata` Action

Extracts Open Graph and meta tags only. Works even for JS-rendered pages because OG tags are in the initial HTML.

```bash
curl -X POST "https://audos.com/api/hooks/execute/workspace-351699/web-api" \
  -H "Content-Type: application/json" \
  -d '{"action": "metadata", "url": "https://example.com"}'
```

**Response:**
```json
{
  "success": true,
  "url": "https://example.com",
  "title": "Example Page",
  "openGraph": {
    "og:title": "Example Page",
    "og:description": "A description",
    "og:image": "https://example.com/image.jpg"
  },
  "meta": {
    "description": "Page description",
    "author": "John Doe"
  }
}
```

## JS-Rendered Page Detection

The enhanced `fetch` action now detects when a page is a Single Page Application (SPA) that renders content via JavaScript.

### How It Works

1. Fetch the raw HTML
2. Strip HTML tags to get text content
3. Compare sizes:
   - If `rawLength > 5000` but `contentLength < 500` → Likely JS-rendered
4. Set `isJsRendered: true` and add a warning

### Example Response for JS-Rendered Page

```json
{
  "success": true,
  "url": "https://www.trythroughline.com",
  "title": "Throughline",
  "content": "Throughline",
  "contentLength": 11,
  "rawLength": 28248,
  "isJsRendered": true,
  "warning": "This page appears to be JavaScript-rendered (SPA). The content is loaded dynamically and cannot be extracted via server-side fetch. Try the 'metadata' action to get Open Graph tags instead."
}
```

### Workaround for JS-Rendered Pages

When you encounter a JS-rendered page:

1. **Use `metadata` action**: Open Graph tags are in the initial HTML and work
2. **Try alternative URLs**: Some sites have server-rendered blog pages even if the main site is a SPA
3. **Use external APIs**: For LinkedIn, Twitter, etc., use their official APIs
